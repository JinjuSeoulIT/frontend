# Reception Display 연동 가이드 (진료 코드 수정 시점용)

현재 `/reception/display`는 조회 전용 화면입니다.  
호출 버튼/호출 처리 로직은 포함하지 않았습니다.

## 목표
- 진료 화면에서 "호출" 동작이 일어나면, 전광판이 자동으로 `WAITING -> CALLED` 상태를 반영

## 권장 방식
1. 진료 화면에서 접수 상태 변경 API 호출
2. 백엔드에서 해당 접수의 `status = CALLED` 저장
3. 전광판은 주기 조회(현재 8초) 또는 실시간 채널(SSE/WebSocket)로 상태 변경 반영

## 프론트(진료 화면) 수정 포인트
1. 접수 선택 후 호출 버튼 클릭 시 `PATCH /api/receptions/{receptionId}/status` 호출
2. 요청 바디 예시
```json
{
  "status": "CALLED",
  "reasonText": "진료 호출"
}
```
3. 호출 성공 시 현재 진료 목록도 재조회하여 화면 동기화

## 백엔드 확인 포인트
1. `CALLED` 상태 전이 허용 여부
2. 상태 변경 이벤트 발행 가능 여부 (실시간 확장 시)
3. 응답 포맷이 현재 프론트의 `ApiResponse` 규격과 일치하는지 확인

## 전광판 변경 필요 여부
- 주기 조회만 사용할 경우: 전광판 추가 수정 불필요
- 실시간 반영으로 바꿀 경우:
  - `/reception/display`에서 polling 제거
  - SSE/WebSocket 수신 시 해당 `receptionId` 상태만 갱신

## 테스트 체크리스트
1. 진료 화면에서 호출 클릭
2. 백엔드 DB에서 해당 접수 상태가 `CALLED`로 변경되는지 확인
3. 전광판에서 해당 환자 상태가 `호출`로 반영되는지 확인
4. 네트워크 실패 시 재시도/오류 안내가 정상인지 확인
